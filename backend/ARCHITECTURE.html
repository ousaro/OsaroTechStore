Core Philosophy

Three rules. Everything follows from them.
01
Dependencies point inward only

Infrastructure depends on domain. Domain depends on nothing. The domain layer has zero imports from any other project layer — only from the shared kernel.

02
Modules communicate via events only

No module imports another module's domain. Cross-module communication happens through domain events on the event bus, mediated by collaboration translators.

03
One composition root owns all wiring

configureApplicationModules.js is the only file that knows the full dependency graph. Nothing else creates or holds module instances.

04
Ports define contracts, adapters implement them

Every external dependency (DB, payment gateway, event bus) is accessed through a port interface. Swap the adapter without touching any use case.

Module Anatomy

Every module has the same internal structure. No exceptions.
composition.js — pure factory, no singletons

The only public file that wires a module. Returns { createRoutes, [collaborationMethod] }. Never imports env. Never uses module-level let exports. The composition root calls it once and holds the instance.

modules/orders/composition.js — canonical patternjs
export const createOrdersModule = ({ orderRepository, orderEventPublisher, logger }) => {
  // 1. Validate output ports at wiring time
  assertOrderRepositoryPort(orderRepository);
  assertOrderEventPublisherPort(orderEventPublisher);

  // 2. Build use cases (builder pattern, closed over deps)
  const addOrder = buildAddOrderUseCase({ orderRepository, orderEventPublisher, logger });

  // 3. Wire input ports
  const commandPort = createOrdersCommandPort({ addOrder, ... });

  // 4. Wire HTTP adapter
  const controller = createOrdersHttpController({ commandPort, queryPort });

  // 5. Return route factory + any collaboration methods
  return {
    createRoutes: ({ requireAuth } = {}) => createOrdersRoutes({ controller, requireAuth }),
    confirmOrderPayment,  // ← consumed by event translator in composition root
  };
};

Composition Root

The only place that sees the full graph
configureApplicationModules.js — execution sequence: 1. Create logger ↓ 2. Resolve providers resolveDatabaseStrategy · resolvePaymentStrategy · resolveEventBus ↓ 3. Connect database database.connect() → dbClient = database.getConnection() ↓ 4. Create repositories (composition root wires them — NOT the DB provider) createMongooseOrderRepository({ dbClient }) createMongoosePaymentRepository({ dbClient }) ↓ 5. Create modules (pure factories, hold instances locally) ordersModule = createOrdersModule({ orderRepository, orderEventPublisher: eventBus, logger }) paymentsModule = createPaymentsModule({ paymentGateway, paymentRepository, ... }) ↓ 6. Register workflows (event bus subscriptions — cross-module collaboration) eventBus.subscribe("OrderPlaced", → paymentsModule.linkPaymentToOrder) eventBus.subscribe("CategoryDeleted", → productsModule.removeProductsByCategory) eventBus.subscribe("PaymentConfirmed", → ordersModule.confirmOrderPayment) ↓ 7. Return to createApp (route factories only — no module refs escape) authRoutes · usersRoutes · productsRoutes · categoriesRoutes · ordersRoutes · paymentsRoutes
⚠

The singleton anti-pattern that was fixed: Original code used export let getAllOrdersHandler; at module level and populated it via configureOrdersModule(). This created mutable globals, made tests non-isolatable, and crashed when import order shifted. The fix: composition root holds all instances in local scope. Nothing else.

Event-Driven Cross-Module Communication

Modules never import each other. Events are the only bridge.
Order placement flow — full event chain: HTTP POST /api/orders ↓ ordersHttpController.addOrder ↓ ordersCommandPort.addOrder ↓ buildAddOrderUseCase → creates Order entity → validates → saves → publishes ↓ OrderPlaced { id, type, occurredAt, version, payload: { orderId, orderLines, ... } } ↓ eventBus.publish() Promise.allSettled([...handlers]) ← handlers isolated, one failure doesn't block others ↓ orderPlacedPaymentLinkTranslator.publish(event) ↓ ACL: assertApplicationEvent() validates envelope paymentsModule.linkPaymentToOrder({ orderId, orderLines }) ↓ buildLinkPaymentToOrderUseCase → creates Stripe session → saves PaymentWorkflow ↓ Stripe webhook later: PaymentConfirmed { orderId, paymentStatus: "paid" } ↓ paymentConfirmedOrderSyncTranslator → ordersModule.confirmOrderPayment ↓ applyPaymentOutcomeToOrder → Order status: pending → processing

Ports & Adapters Pattern

Ports define what. Adapters define how. Composition root decides which.
Output port: assertOrderRepositoryPort

Validates that the injected repository implements the required methods at startup. If a method is missing, you get a clear error at boot — not a cryptic undefined is not a function at runtime.

ports/output/ordersOutputPort.jsjs
const REPO_METHODS = ["findAll", "findById", "create", "updateById", "deleteById"];

export const assertOrderRepositoryPort = (repo) => {
  for (const m of REPO_METHODS) {
    if (typeof repo[m] !== "function")
      throw new Error(`orderRepository must implement .${m}()`);
  }
  return repo;
};
→

To add a PostgreSQL repository: Create modules/orders/adapters/output/repositories/postgres/pgOrderRepository.js that exports createPgOrderRepository({ dbClient }) with the same methods. Add the "postgres" case to resolveDatabaseStrategy.js. Set DATABASE_PROVIDER=postgres. Done — zero module code changes.

How to Add a New Database

Example: adding PostgreSQL with pg/Knex
Step 1 — Create the providerjs
// src/infrastructure/providers/databases/postgres/postgresProvider.js
import knex from "knex";

export const createPostgresProvider = ({ connectionString, logger }) => {
  const db = knex({ client: "pg", connection: connectionString });

  return {
    connect:       async () => { await db.raw("SELECT 1"); logger.info({ msg: "PostgreSQL connected" }); },
    disconnect:    async () => db.destroy(),
    getConnection: ()    => db,   // same interface as mongoProvider
  };
};
Step 2 — Add to resolverjs
// resolveDatabaseStrategy.js — add one case
case "postgres":
  return createPostgresProvider({ connectionString: env.postgresUrl, logger });
Step 3 — Create repository adapter per modulejs
// modules/orders/adapters/output/repositories/postgres/pgOrderRepository.js
export const createPgOrderRepository = ({ dbClient }) => ({
  findAll:    () => dbClient("orders").select("*").orderBy("created_at", "desc"),
  findById:   (id) => dbClient("orders").where({ id }).first(),
  create:     (data) => dbClient("orders").insert(data).returning("*"),
  updateById: (id, data) => dbClient("orders").where({ id }).update(data).returning("*"),
  deleteById: (id) => dbClient("orders").where({ id }).delete().returning("*"),
});
✓

Composition root update: In configureApplicationModules.js, the repository creation is already isolated per module. Wrap it in a DATABASE_PROVIDER check to pick the right factory. No module code changes anywhere else.

Patterns Quick Reference

The building blocks used throughout this codebase
Builder Pattern for Use Cases

buildAddOrderUseCase({ repo, logger }) returns a function. Dependencies are closed over. Easy to test: pass mocks as arguments.

Anti-Corruption Layer (ACL)

Collaboration translators (orderPlacedPaymentLinkTranslator) validate and translate events between bounded contexts. Modules stay independent.

Strategy Pattern

Each resolver (resolveDatabaseStrategy, resolvePaymentStrategy, resolveEventBus) selects an adapter via config. Open for extension, closed for modification.

Record Mapper

Every repository has a mapper (toOrderRecord). DB format ↔ domain primitives conversion happens exclusively in the mapper. Domains never see snake_case or MongoDB ObjectIds.

Port Assertion

assertOrderRepositoryPort(repo) validates the contract at wiring time. Fail fast at startup rather than fail mysteriously at runtime on first use.

Domain Event Envelope

createDomainEvent(type, payload) stamps every event with id (dedup), occurredAt (timeline), version (schema evolution), correlationId (tracing).
